# Security Policy

## Supported Versions

Only the latest version of this project is actively supported with security updates.

| Version        | Supported |
| -------------- | --------- |
| Latest         | Yes       |
| Older versions | No        |

## Reporting a Vulnerability

If you discover a security vulnerability, please report it privately.

Do not open a public GitHub issue for security problems.

Please send a report to:

**[security@example.com](mailto:Zero2Love@protonmail.com)**

Replace this email with your real security contact before publishing.

Include as much detail as possible, such as:

* A clear description of the vulnerability
* Steps to reproduce the issue
* The affected files, routes, commands, or services
* The possible impact
* Any proof of concept, screenshots, logs, or error messages
* Your contact information if you want follow-up

## Response Expectations

After receiving a valid report, I will try to:

* Confirm receipt of the report
* Investigate the issue
* Fix the vulnerability as soon as reasonably possible
* Release a patch or mitigation
* Credit the reporter if they want to be credited

Response times may vary because this is a personal project.

## Responsible Disclosure

Please give me reasonable time to investigate and fix the issue before publicly disclosing it.

Do not use the vulnerability to:

* Access, modify, or delete data that does not belong to you
* Disrupt services
* Exfiltrate secrets, tokens, passwords, or private data
* Perform attacks against users or third-party systems
* Spam, phish, or socially engineer anyone

## Scope

Security reports are welcome for vulnerabilities in this repository, including:

* Authentication or authorization issues
* Exposed secrets or tokens
* Remote code execution
* Server-side request forgery
* Cross-site scripting
* SQL injection
* Command injection
* Insecure file uploads
* Path traversal
* Sensitive data exposure
* Dependency vulnerabilities with real impact

## Out of Scope

The following are generally out of scope:

* Reports from automated scanners without proof of real impact
* Denial-of-service attacks
* Social engineering
* Physical attacks
* Issues requiring access to a user’s own compromised device
* Vulnerabilities in third-party services unless this project directly misuses them
* Missing security headers without a practical exploit
* Low-impact best-practice suggestions without a real security risk

## Secrets and Credentials

If you find exposed credentials, API keys, tokens, or private configuration, report them immediately.

Do not test, use, or share exposed secrets.

## Public Disclosure

Please do not publicly disclose a vulnerability until a fix or mitigation has been released.

If credit is given, it will be added in the release notes, changelog, or security advisory unless you request otherwise.

## Safe Harbor

I will not take legal action against good-faith security research that follows this policy, avoids privacy violations, does not disrupt services, and gives me reasonable time to fix the issue.
